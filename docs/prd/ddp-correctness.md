# DDP-Correctness Fix -- PRD

## Background

The four training CLIs (`manifold-train`, `manifold-train-paired`, `manifold-train-reward`, `manifold-train-grpo`) disagree about what `-g 1` means and about how multi-GPU is detected, and the three generative CLIs carry several classes of DDP defect. Verified defect classes:

1. **Monitor-default bug (JiT + Paired only).** `cli.py:144` and `paired_cli.py:145` compute `multi_gpu = isinstance(devices, int) and devices > 1`. For `devices="auto"` on a multi-GPU box this is `False`, so `monitor_fid` (`cli.py:170`) / `monitor_psnr` (`paired_cli.py:148`) stay `True` while `build_trainer` (`trainer.py:79`) spawns DDP. `ModelCheckpoint` then monitors a rank-0-only metric under DDP. GRPO already mirrors `build_trainer` (`grpo_cli.py:161-163`); reward is incidentally safe (`reward_cli.py:268` passes `devices=1`).
2. **Surprise `-g1 -> auto -> DDP`.** `cli.py:325` and `paired_cli.py:250` fall back to `devices="auto"` whenever `num_gpus <= 1`, so `-g 1` (or no `-g`) on a multi-GPU host silently becomes DDP. reward/grpo already pass `1`.
3. **Pre-PG warm + dishonest metrics.** The VAE-encode warm (`cli.py:352`, `paired_cli.py:356/367`) runs in `main()` before the process group exists, so it cannot shard per-rank. Linear metrics (`metrics.py:49` `train/loss_epoch`, `metrics.py:82` `val/x0_mae`) log `self._sum/self._n` with **no `batch_size` and no `sync_dist`** — a per-rank mean-of-batch-means mislabeled as global. `train/loss` (`grpo.py:502`) and `train/grad_norm` (`latent_flow.py:376`, `paired_latent_flow.py:319`) need separate analysis. GRPO `validation_step` (`grpo.py:540-563`) generates + scores on all ranks. `feature_net`/`real_latents` (`cli.py:303`, `cli.py:362`, `grpo_cli.py:521/525`) are built on every rank pre-PG. Rank-0-only metrics (`psnr_ssim_callback.py:265`) and their warnings (`psnr_ssim_callback.py:114`, `fid_callback.py:127`) are unlabeled / fire on all ranks.

**Two load-bearing invariants verified this review (must be preserved, not broken by the refactor):**
- `build_trainer` sets `find_unused_parameters=True` (`trainer.py:79`, comment at `trainer.py:73-75`): DDP correctness of the **paired** path depends on it, because the `_PinnedClassEmbedding` shim (PR #70) means a class-embedding row can receive no gradient when labels are dropped. Any DDP-related edit or the warm refactor MUST keep this flag.
- `DoubleEMACallback` (`ema.py:4-7`) asserts "DDP all-reduces the params first, so the [double-EMA] shadows stay in sync with no extra collective." This keeps the published slow-EMA shadow (used for rank-0 FID generation) numerically equal across ranks — but it is currently an unverified docstring claim.

**GRPO training has a DDP hazard that is out of scope to FIX and must instead be SCOPED OUT.** The PPO inner loop (`grpo.py:484-500`) runs `K = len(eta_step_list)` iterations, each `self.manual_backward(loss)` → `opt.step()` → `opt.zero_grad(set_to_none=True)`, with zero `no_sync()` usage (grep confirms 0 occurrences). Combined with `build_trainer`'s `find_unused_parameters=True`, this loop is unsafe under DDP (each backward re-traverses the autograd graph; the async all-reduce fires K times racing `zero_grad`). The loop length is config-driven and identical across ranks, so this is a backward-semantics hazard, not a divergence hazard. GRPO v1 ships single-GPU (euler PID 1398415); the multi-backward DDP fix (`no_sync` on all-but-last inner backward, or `static_graph=True` + drop `find_unused_parameters`) is non-trivial and out of scope.

**DDP seeding policy (no code change; documents the invariant the items below depend on).** All four CLIs call `pl.seed_everything(seed, workers=True)` on every rank with the SAME seed (`cli.py:270`, `paired_cli.py:219`, `grpo_cli.py:296`, `reward_cli.py:232`). Consequences: (a) model init is bit-identical across ranks (intended); (b) training noise is correlated across ranks — a variance characteristic, not a correctness bug, and data diversity is provided by Lightning's `DistributedSampler` (per-rank shard offset), which **must** be attached to the val loader for M6/M5 to mean anything; (c) generation noise is shared across ranks, tolerated because generation is rank-0-only.

The repo just landed an **unrelated LR-scaling feature** (`latent_flow.py` / `paired_latent_flow.py` `configure_optimizers` + `lr_ref_batch_size`/`lr_scale_rule`/`lr_warmup_ratio` kwargs). Verified **non-conflicting**: no LR edit touches any DDP finding's file:line, and `len(latent_ds) == len(vol_ds)` (`latent_dataset.py:105-106`), so the LR horizon is unchanged by the warm-move refactor.

Locked decisions (ADR-0016 / ADR-0017): **(A)** keep rank-0-only FID/PSNR; fix the monitor guard + add honest cross-rank reduction for linear metrics; do NOT build distributed FID/PSNR (offline export + `save_last`/`every_n_epochs` covers selection). **(B)** move the VAE-encode warm into a Lightning `DataModule.setup()` (post-PG, per-rank) for the two encoding CLIs only; reward/grpo load pre-computed `.pt` latents and are unaffected. **(M2)** unify `-g 1 -> devices=1`; multi-GPU requires explicit `-g N`.

## Goals

- Multi-GPU runs of the **JiT and Paired** CLIs are correct and honest: no phantom `monitor=` that breaks `ModelCheckpoint`, no surprise `auto -> DDP` from `-g1`, linear metrics reduced to the **true sample-weighted global mean**, rank-0-only metrics explicitly scope-labeled, VAE warm sharded one-per-rank.
- **GRPO training is single-GPU by construction** (multi-GPU explicitly blocked with a clear error); its `validation_step` rank-0 gate (M3) still ships as an honesty/defensive measure.
- Zero functional change to the single-GPU path (the only config shipped: euler, gauss, sugon) and to the LR-scaling feature — gated by an automated single-GPU regression test.
- Two-PR split: mechanical sweep first (low-risk, no data-path refactor), warm refactor second.

## Non-Goals

- No distributed FID/PSNR (cross-rank generation + all-gather) -- rejected by Decision A; offline export + `save_last`/`every_n_epochs` covers selection.
- No async eval-decoupling (rank 1..N-1 idle during rank-0 generation is tolerated, not a deadlock) -- deferred by Decision A.
- **No multi-GPU GRPO *training*** (the `no_sync`/`static_graph` fix for the PPO inner loop) -- out of scope; GRPO is blocked to `devices=1` instead (G2). GRPO `validation_step` rank-0 gate (M3) stays.
- No changes to `reward_cli.py` device/monitor behavior beyond items listed below (already correct: G1, R1).
- No edits to the LR-scaling feature (orthogonal; verified non-conflicting).
- No re-architecture of the warm path beyond the atomic `DataModule.setup()` move + lazy `FIDCallback.real_latents` plumbing.

## Phasing

- **Prerequisite task (blocks every "2-rank smoke" gate).** Stand up the `ddp_cpu_two_rank` pytest fixture (Lightning `DDPStrategy` with `gloo`, via `torchrun --nproc-per-node=2`/spawn under CPU) and land it first. Pin it once; every multi-rank criterion references it by name.
- **PR #1 -- mechanical correctness sweep (no data-path refactor).** Lands: **SG1** (standing single-GPU gate), **G2** (block multi-GPU GRPO), **M1a, M1b** (`is_multi_gpu()` helper), **M2a, M2b** (`-g1→1`), **M3** (val rank-0 gate + no-deadlock gate), **M4**, **M6** (`MeanMetric` migration), **D1** (val-loader sampler verification), **L5**, **G3** (grad_norm parity), **M5**, **L3** (lazy `feature_net`), **L1**, **E1** (EMA parity test). ## Risks & Mitigations

- **Shared-accessor correctness (M1a/M1b).** `is_multi_gpu()` reuses `trainer.py:110 __cuda_device_count()` (which guards `is_available()`); an inline `device_count()>1` without the guard could diverge under `CUDA_VISIBLE_DEVICES`. Mitigation: one accessor, all three sites; unit test asserts equivalence under device-mask permutations. Confirmed side-effect (closed): on `devices=1` or `"auto"` on a 1-GPU box the new expression is `False`, so `monitor_fid`/`monitor_psnr` stay `True` and single-GPU selection is preserved.
- **Naive `sync_dist` is NOT the global mean (M6).** Decision A literally says "add `sync_dist`"; verified that gives a mean-of-per-rank-means, exact only under equal per-rank batch counts (which DDP padding happens to guarantee today, but would silently break under a non-padding sampler / `drop_last=True` / asymmetric `limit_val_batches`). Mitigation: M6 migrates to `torchmetrics.MeanMetric` (true sample-weighted reduction) to actually satisfy Decision A's honesty intent; the unbalanced-shard sub-assertion locks the property. Flagged for an ADR-0016 amendment in Open Questions.
- **`val/mean_reward` regime shift (M3/M4).** Rank-0 gating makes the denominator a single shard. Mitigation: the metric was always a per-shard mean mislabeled as global; the fix only makes the label honest. GRPO v1 ships single-GPU (and G2 blocks multi-GPU training), so no `-g1` baseline regression; document in the ADR/changelog.
- **GRPO multi-backward DDP hazard (G2).** The PPO inner loop (`grpo.py:484-500`) is unsafe under `find_unused_parameters=True` + K backwards. Mitigation: G2 blocks `num_gpus>1` so the hazard is never exercised; M3's val gate is harmless under single-GPU. The `no_sync`/`static_graph` fix is a Non-Goal.
- **`find_unused_parameters=True` preservation.** The paired class-embedding shim (PR #70) relies on it (`trainer.py:73-75`); the warm refactor must not move model construction or drop the flag, or the paired CLI deadlocks on its first multi-GPU run. Mitigation: F1/F2/F4 touch only the warm/datamodule path, not `build_trainer`; called out here so a reviewer does not remove it.
- **EMA-stays-in-sync invariant (E1).** The published slow-EMA shadow is only valid if DDP's param all-reduce happens before `on_train_batch_end`'s shadow update on every rank. Mitigation: E1 is a one-shot 2-rank `torch.equal` assertion; if it fails, the EMA callback needs an explicit cross-rank sync (new finding).
- **Cold-cache `scale_factor` drift (F1/F2).** Sharding the warm changes the population over which `estimate_scale_factor` runs. Mitigation: codex-verified `estimate_scale_factor` is deterministic and sampled rank-0 with seed 0; `val_latents` is rank-constant (F1 assertion). Verify `val/fid` is unchanged on a warm-cache run (cache hits → no re-encode) before claiming cold-cache parity. This is the one metric to watch on first 2-rank cold start.
- **Cross-rank cache write races (F2/F4).** Per-rank `setup()` writes to a shared cache dir. Mitigation: the sharded branch (F2/F3) gives one writer per shard/unique-volume cache file; NCCL barrier after warm; 2-rank cold+warm smoke asserts cache integrity.
- **FIDCallback lazy-real_latents ordering (F5).** If `real_latents` is not populated before the first `_real_features` call, FID crashes. Mitigation: F5 test asserts non-None before the first val epoch + single-GPU `val/fid_avg` regression.
- **`-g1` behavior change (M2a/M2b).** Users relying on the old `auto` fallback now get a single device. Mitigation: M2 is a locked decision and reward/grpo already behave this way; pre-merge grep of euler/gauss/sugon launch commands confirms no shipped invocation relies on silent auto-DDP; note in the changelog.
